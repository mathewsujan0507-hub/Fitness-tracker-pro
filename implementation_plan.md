# Migrate from Frontend-Only to Full-Stack Node.js App

This plan will convert the current `Fitness-tracker-pro` project from a standalone HTML/CSS/JS prototype into a fully functional full-stack web application using Node.js, Express, and an SQLite database. It will secure the user authentication system and establish a backend for saving fitness data securely per user.

## User Review Required

> [!WARNING]
> Because you are converting a frontend app to a full-stack app, this requires changes across multiple files, structural shifts, and replacing the `localStorage` database with real HTTP `fetch()` requests and an SQLite database.

Are you okay with using **Node.js, Express, and SQLite3** for the backend, as this is lightweight and ideal for this type of application? 

## Proposed Changes

---

### Backend Server Structure

#### [NEW] package.json
Initialize a Node.js project. We will install:
- `express` (web server)
- `sqlite3` + `sqlite` (database driver)
- `bcrypt` (secure password hashing)
- `express-session` (handling user sessions via cookies)

#### [NEW] server.js
The entry point for the backend. It will:
1. Serve the current directory as static files via Express.
2. Initialize the SQLite database connection.
3. Define the REST API routes.

#### [NEW] database.sqlite
An SQLite database file that will be automatically generated by the server.

---

### Database Schema (SQLite)

We will create the following tables on server startup:
- `users`: `id` (PK), `email` (UNIQUE), `password` (hashed)
- `metrics`: `id` (PK), `user_id` (FK), `water`, `steps`, `total_workouts`, `total_time`
- `plans`: `id` (PK), `user_id` (FK), `data` (JSON string of plan content)
- `history`: `id` (PK), `user_id` (FK), `data` (JSON string of workout history)

---

### REST API Endpoints

I will implement the following routes in `server.js` to replace the `localStorage` access:

- `POST /api/auth/signup` - Registers a new user with a hashed password
- `POST /api/auth/login` - Logs a user in and establishes an HTTP cookie session
- `POST /api/auth/logout` - Destroys the session securely
- `GET /api/auth/me` - Verifies if the user is logged in
- `GET | POST | PUT /api/plans` - CRUD operations for workout plans
- `GET | POST /api/dashboard` - Fetches and updates dashboard metrics (water/steps)
- `GET | POST /api/history` - Fetches and saves workout histories

---

### Frontend Modifications

To connect the beautiful frontend you already have to the new Node.js server, we will rewrite the data-storing functions. Instead of using `localStorage`, the web app will use Javascript `fetch()` API calls to interact with the backend routes.

#### [MODIFY] js/auth.js
Change `signup()` and `login()` functions to make POST requests to `/api/auth/...` instead of reading/writing to `localStorage("ft_users")`.

#### [MODIFY] js/dashboard.js
Extract water, steps, total time, and total workouts via a GET request to `/api/dashboard`.

#### [MODIFY] js/planner.js & js/library.js
Update the saving logic to persist plans via the `/api/plans` routes.

#### [MODIFY] js/session.js & js/history.js
Update the workout completion and history fetching logic to use `/api/history`.

## Open Questions

1. **Authentication Mode**: Currently, the frontend uses `localStorage` to manage users globally without real security. This plan will switch to secure **cookies/sessions** (`express-session`). Are you okay with that?
2. **Current Local Data**: Because we are migrating to a real database, any user data currently stored in your browser's local storage during your prototype testing will NOT carry over automatically. Is this a fresh start?

## Verification Plan

### Automated Tests
1. Verify `server.js` starts cleanly without database connection errors.
2. Ensure API routes respond correctly via local browser requests.

### Manual Verification
1. Run the server using `node server.js` and open the app via `http://localhost:3000`.
2. Register a new user and ensure they appear in the SQLite database correctly (with hashed passwords).
3. Log out and log back in.
4. Interact with the dashboard (add water, track a session) and refresh the page to verify that data survives across page reloads via the backend SQLite database.
